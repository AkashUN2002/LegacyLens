# Multi-Modal Evidence Review Pipeline

An AI-powered insurance claim review system that uses a LangGraph workflow to evaluate claims against photo evidence. It processes a batch of claims from a CSV, sends each through a multi-step reasoning pipeline backed by Amazon Bedrock (Claude), and writes a structured verdict CSV.

## Dataset Folder

The `dataset/` folder must live **one level above** `code/` (i.e., as a sibling). The pipeline resolves all paths relative to this layout automatically.

| File | Description |
|---|---|
| `claims.csv` | **Test set input.** 4 columns only: `user_id`, `image_paths`, `user_claim`, `claim_object`. No ground-truth labels. Images reference `images/test/case_XXX/`. Run the pipeline against this file to produce predictions. |
| `sample_claims.csv` | **Labeled sample set.** All 14 columns (input + expected output). Images reference `images/sample/case_XXX/`. Used both as a pipeline input for development runs and as the ground-truth source for evaluation. |
| `evidence_requirements.csv` | Rules table with columns `requirement_id`, `claim_object`, `applies_to`, and a requirement description. Used by the `evidence_check` node. |
| `user_history.csv` | One row per `user_id` with a `history_flags` column (e.g. `user_history_risk;manual_review_required` or `none`). |
| `output.csv` | Written by `main.py` after a run. Contains the 14-column predictions. |
| `images/` | Image files referenced in the `image_paths` column of the CSVs. Paths are semicolon-separated and relative to the `dataset/` root. Two splits: `images/sample/case_001/img_1.jpg` (for sample runs) and `images/test/case_001/img_1.jpg` (for test runs). Case folders are zero-padded three-digit numbers (`case_001`, `case_002`, …). |

## Setup

### 1. Install dependencies

```bash
cd code
pip install -r requirements.txt
```

### 2. Configure environment variables

Create a `.env` file inside the `code/` directory:

```env
ANTHROPIC_MODEL=us.anthropic.claude-sonnet-4-5-20251203-v1:0
VISION_MODEL=us.anthropic.claude-opus-4-8-20260101-v1:0
AWS_ACCESS_KEY_ID=your_access_key
AWS_SECRET_ACCESS_KEY=your_secret_key
AWS_DEFAULT_REGION=us-east-1
```

- `ANTHROPIC_MODEL` — Used for `extract_claim` and `reconcile` nodes (text reasoning).
- `VISION_MODEL` — Used for `analyze_images` node (multimodal vision). Falls back to `ANTHROPIC_MODEL` if not set.

Alternatively, AWS credentials can be provided via standard AWS configuration files or IAM roles.

## Running the Pipeline

### Run on the labeled sample set (default)

Uses `dataset/sample_claims.csv` → `dataset/output.csv`:

```bash
cd code
python main.py
```

### Run on the test set

```bash
python main.py ../dataset/claims.csv ../dataset/output.csv
```

### Custom input/output paths

```bash
python main.py path/to/input.csv path/to/output.csv
```

Progress is printed per row:

```
[1/20] user_001 car      -> supported (valid_image=true)
[2/20] user_002 laptop   -> not_enough_information (valid_image=false)
...
Wrote 20 rows to .../dataset/output.csv
```

## Output Format

`output.csv` contains 14 columns:

| Column | Description |
|---|---|
| `user_id` | Echoed from input |
| `image_paths` | Echoed from input |
| `user_claim` | Echoed from input |
| `claim_object` | `car`, `laptop`, or `package` |
| `evidence_standard_met` | `true` / `false` |
| `evidence_standard_met_reason` | Human-readable explanation |
| `risk_flags` | Semicolon-separated flags (e.g. `blurry_image;manual_review_required`) |
| `issue_type` | e.g. `dent`, `crack`, `glass_shatter`, `none`, `unknown` |
| `object_part` | e.g. `front_bumper`, `screen`, `seal`, `unknown` |
| `claim_status` | `supported`, `contradicted`, or `not_enough_information` |
| `claim_status_justification` | 1–2 sentence reasoning |
| `supporting_image_ids` | Semicolon-separated image IDs that grounded the verdict |
| `valid_image` | `true` / `false` |
| `severity` | `none`, `low`, `medium`, `high`, or `unknown` |

## Workflow Architecture

The pipeline is a LangGraph state machine with the following nodes:

```
START
  └─► load_context         (deterministic: parse images, normalize claim_object)
        └─► extract_claim  (LLM text: extract structured claim from transcript)
              └─► analyze_images  (LLM vision: per-image findings, usability)
                    └─► [usable_gate]
                          ├─► evidence_check  (deterministic: coverage check)
                          │     └─► reconcile (LLM text: supported/contradicted/NEI)
                          ├─► reconcile       (unusable but informative path)
                          └─► force_nei       (deterministic: truly unusable)
                    └─► risk_merge            (deterministic: union all flags)
                          └─► finalize        (deterministic: coerce + emit output row)
                                └─► END
```

**LLM nodes (3 per claim on the normal path):**

| Node | Model | Purpose |
|---|---|---|
| `extract_claim` | Sonnet | Parses transcript into structured claim; detects manipulation attempts |
| `analyze_images` | Opus (vision) | Produces per-image findings without knowing the claim |
| `reconcile` | Sonnet | Compares claim vs findings to decide verdict |

All other nodes are deterministic (no model calls). If all images are unusable and non-informative, `force_nei` short-circuits, skipping `reconcile` (2 LLM calls instead of 3).

## Evaluation

After running the pipeline on `sample_claims.csv`, score predictions against ground truth:

```bash
python evaluation/main.py
```

Or with explicit paths:

```bash
python evaluation/main.py path/to/output.csv path/to/sample_claims.csv
```

The evaluation reports:

- **Categorical field accuracy** for: `evidence_standard_met`, `valid_image`, `claim_status`, `issue_type`, `object_part`, `severity`
- **Set-based accuracy** for: `risk_flags`, `supporting_image_ids`
- **claim_status confusion matrix**
- **Severity error profile** (off-by-one vs off-by-more)
- **Free-text completeness** for justification fields
- A `metrics.json` file written to `evaluation/` with machine-readable scores

---

## Error Handling

If any single claim raises an exception during processing, it is caught and a safe `not_enough_information` row is emitted with `risk_flags=manual_review_required`. The batch always produces a complete `output.csv`.

## Key Design Notes

- **Image normalization:** images are sniffed by magic bytes (not file extension, which is unreliable in the dataset). Oversized or unrecognized formats are decoded with Pillow, downscaled to a 2,048 px long edge, and re-encoded as JPEG under 4.5 MB before sending to Bedrock.
- **Injection resistance:** both the image analysis and claim extraction prompts treat all embedded text as data, never as instructions. Manipulation attempts are flagged in `risk_flags`.
- **Model routing:** Opus is used only for the vision node where it is needed; Sonnet handles both text nodes. This is both cheaper and more accurate than using Opus throughout.
- **Retry/reliability:** the Bedrock client uses a 120s read timeout and adaptive retry (up to 4 attempts) to handle transient throttling.
